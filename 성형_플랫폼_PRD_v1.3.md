## 1. 서비스 개요

### 1.1 서비스 비전

성형을 고민하는 사람들이 정보의 불균형 없이 신뢰할 수 있는 커뮤니티 안에서 실제 경험을 공유하고, 필요 시 병원과 자연스럽게 연결되는 플랫폼.

### 1.2 핵심 문제 정의

- 성형 시장의 정보 불균형 — 실제 후기보다 병원 마케팅 콘텐츠에 과다 노출
- 커뮤니티 파편화 — 카페·카톡·인스타 등 정보 채널 분산
- 비포&애프터 정보 부족 — 병원 직접 게시 불가(의료광고 규제)로 인한 정보 공백
- 병원 연결 과정의 불투명성 — 가격·시술 정보 비교 어려움

### 1.3 목표 사용자

| 구분 | 특성 | 핵심 니즈 |
|------|------|----------|
| 일반 유저 (소비자) | 성형 고민 중인 20–40대 여성 중심 | 신뢰할 수 있는 실제 후기, 병원 정보, 커뮤니티 |
| 병원 (B2B) | 성형외과 원장/마케터 | 잠재 고객 확보, 정보 노출 |
| CMS 관리자 | 플랫폼 운영팀 | 콘텐츠 검수, 유저/병원 관리, 신고 처리 |

### 1.4 플랫폼 포지셔닝

**성형 정보·커뮤니티 플랫폼**으로 포지셔닝한다.  
역경매 상담 연결 기능은 포함되어 있으나, 수익 모델은 추후 법률 검토 후 별도 확정한다.  
커뮤니티와 콘텐츠가 서비스의 핵심 가치이며, 병원 연결은 부가 기능으로 위치한다.

## 2. 역경매 상담 요청 시스템

> ⚠️ 기능은 포함하되, **수익 모델(병원 과금 구조)은 법률 검토 후 별도 확정**. 현재는 무과금 구조로 개발 진행.  
> 의료법 제27조 관련 법률 자문은 출시 전 필수.

### 2.1 구조 개요

유저가 시술 고민을 등록하면 병원들이 열람 후 상담 메시지를 발송하는 역경매 구조.  
1개 요청당 최대 **5개 병원** 선착순 발송. 유저는 수신된 상담을 비교 후 1개 병원을 선택해 1:1 채팅으로 전환한다.

---

### 2.2 유저 상담 요청 플로우

#### Step 1 — 시술 카테고리 선택

| 항목 | 내용 |
|------|------|
| UI | 아이콘 + 텍스트 그리드 (2열) |
| 선택 방식 | 단일 선택 |
| 카테고리 | 눈 / 코 / 윤곽 / 가슴 / 지방흡입 / 피부·레이저 / 기타 |
| 관리 | CMS에서 추가·수정·삭제 |

#### Step 2 — 세부 시술 선택

| 항목 | 내용 |
|------|------|
| UI | 칩(Chip) 멀티셀렉 |
| 선택 방식 | 복수 선택 가능 |
| 예시 (눈) | 쌍꺼풀 매몰법 / 절개법 / 앞트임 / 뒤트임 / 눈밑지방 / 안검하수 교정 |
| 예시 (코) | 콧대 / 코끝 / 들창코 / 매부리코 / 비중격 교정 |
| 관리 | CMS에서 카테고리별 항목 관리 |

#### Step 3 — 사진 촬영 및 업로드

**기본 원칙**
- 보정·필터 완전 차단 — 카메라 UI 필터 옵션 미노출, 원본 업로드
- 앱 내 카메라 실행 (갤러리 업로드 병행 허용, 단 가이드 프레임 미제공)
- 업로드 전 안내: "보정 없는 사진이 정확한 상담에 도움됩니다"
- 촬영 사진은 병원에게만 공개 (커뮤니티 비공개)

**부위별 촬영 가이드 프레임**

앱 내 카메라 실행 시 반투명 실루엣 오버레이 표시. 각도별로 순서대로 촬영.

| 부위 | 촬영 장수 | 각도 순서 | 프레임 형태 |
|------|----------|-----------|------------|
| 눈 | 3장 | ① 정면(눈 뜬) → ② 정면(눈 감은) → ③ 클로즈업(좌 또는 우) | 눈 영역 타원형 |
| 코 | 4장 | ① 정면 → ② 좌측면(90°) → ③ 우측면(90°) → ④ 45도 사선 | 코 중심 직사각형 |
| 윤곽 (턱·광대) | 4장 | ① 정면 → ② 좌측면 → ③ 우측면 → ④ 45도 사선 | 얼굴 전체 윤곽 |
| 가슴 | 3장 | ① 정면 → ② 좌측면 → ③ 우측면 | 흉부 영역 직사각형 |
| 지방흡입 | 3장 | ① 정면 → ② 측면 → ③ 후면 | 해당 부위 영역 |
| 기타 | 3장 | ① 클로즈업 정면 → ② 측면 → ③ 후면 | 자유 프레임 |

**촬영 UX**
- 각도별 한 장씩 순서 안내 (각도명 + 예시 일러스트)
- 완료된 장 하단 썸네일 표시, 재촬영 가능
- 모든 각도 완료 시 다음 Step 자동 이동
- 조명 부족 감지 시 경고 문구 표시

**사진 공개 토글**

| 설정 | 내용 |
|------|------|
| ON (기본값) | 병원이 사진 포함 전체 열람 가능 |
| OFF | 텍스트만 노출, 사진은 1:1 채팅 전환 후 공유 |

#### Step 4 — 상담 요청 내용 입력

| 항목 | 내용 |
|------|------|
| UI | 자유 텍스트 멀티라인 |
| 글자 수 | 최소 20자 / 최대 500자 |
| 추가 항목 | 희망 시술 시기 (1개월 이내 / 3개월 이내 / 6개월 이내 / 미정) |

#### Step 5 — 공개 범위 (MVP)

- MVP: 전체 병원 공개 단일 옵션
- 안내: "등록된 병원들이 요청을 확인하고 상담을 보내드립니다"

#### Step 6 — 최종 확인 및 제출

| 항목 | 내용 |
|------|------|
| UI | 요약 카드 (시술, 사진 썸네일, 요청 내용, 시기) |
| 제출 후 안내 | "72시간 내에 최대 5개 병원에서 상담이 도착할 예정입니다" |
| 요청 유효 기간 | 72시간 |
| 알림 | 상담 도착 시 푸시 알림 |
| 취소 | 72시간 내 취소 가능 (발송된 상담은 유지) |

### 2.4 병원 — 상담 메시지 작성

#### 구조화 필드 (필수)

| 필드명 | 입력 방식 | 설명 |
|--------|----------|------|
| 병원 한 줄 소개 | 텍스트 최대 60자 | 프로필 자동완성, 수정 가능 |
| 전문 시술 분야 | 칩 선택 최대 3개 | 프로필 등록 분야 기반 자동 노출 |
| 해당 시술 경력 | 텍스트 최대 60자 | 예: "쌍꺼풀 절개법 전문, 15년 경력" |
| 상담 가능 방식 | 체크박스 | 방문 / 비대면 화상 / 채팅 |
| 상담 가능 시간 | 텍스트 또는 시간 선택 | 예: "평일 10:00–18:00" |

#### 자유 텍스트 (필수)

- 최소 50자 / 최대 1,000자

#### 선택 항목

| 필드명 | 내용 |
|--------|------|
| 병원 사진 | 최대 3장 (병원 내부·장비만, 시술 전후 불가) |
| 첫 상담 안내 문구 | 최대 100자 |

#### 금액 기재 차단

- **키워드 필터링**: "만원", "원", 숫자+단위 패턴 감지 시 발송 차단 + 경고 문구 표시
- 필터링 우회 시도(한글 변환 등) 감지 로직 추가
- 위반 적발 시: 경고 1회 → 2회 누적 시 계정 정지

#### 발송 제약

| 항목 | 내용 |
|------|------|
| 발송 상한 | 요청당 최대 5개 병원 (선착순) |
| 발송 취소 | 유저 선택 전까지 취소 가능 |
| 수정 | 발송 후 수정 불가 |
| 재발송 | 동일 요청 재발송 불가 |
| 추가 금지 | 외부 연락처 유도 (전화번호, 카카오톡 ID 등) |

## 3. 성형 커뮤니티

### 3.1 비포&애프터 게시판

> ⚠️ 유저 자율 커뮤니티로 운영. 플랫폼은 인센티브를 제공하지 않으며, 자연 발생 콘텐츠 구조로 설계. (의료광고 간접 유도 리스크 해소)

**운영 원칙**
- 유저가 자발적으로 시술 전후 사진 + 후기 게시
- **플랫폼 인센티브 없음** — 포인트·리워드 미제공
- 병원 태깅 허용 (유저 자발적 선택, 강제 아님)
- 태깅된 병원은 해당 게시물 승인·거부 불가 (유저 주도권 유지)
- 신고 기능 → CMS 검수 후 블라인드

**게시물 구성**
- 시술 카테고리 태그 (필수)
- 시술 시기 입력 (선택)
- 전후 사진 (선택, 최대 10장)
- 후기 텍스트 (최소 20자)
- 병원 태깅 (선택)

### 3.2 자유 게시판 / Q&A

- 성형 관련 자유 의견 교환
- 익명 게시 지원
- 태그 기반 분류 (#눈성형, #코성형, #회복기간 등)
- 댓글·대댓글·좋아요 지원

### 3.3 투표 / 설문

- 유저 또는 운영팀이 투표 생성
- 이미지 포함 투표 지원
- 결과 실시간 공개 (참여 후 노출)
- 설문: 복수 선택형 + 주관식 혼합
- 인기 투표 메인 피드 노출 → 커뮤니티 활성화 유도

## 4. 유튜브 콘텐츠 히어로

### 4.1 기능 개요

**플랫폼 운영사가 직접 제작·운영하는 유튜브 채널의 영상**을 메인화면 히어로 영역에 노출.  
병원 콘텐츠가 아닌 운영사 자체 콘텐츠이므로 의료광고 심의 대상 아님.  
터치 시 유튜브 앱(미설치 시 브라우저)으로 이동.

| 항목 | 내용 |
|------|------|
| 콘텐츠 주체 | 플랫폼 운영사 자체 유튜브 채널 |
| 썸네일 수집 | `img.youtube.com/vi/{ID}/hqdefault.jpg` 또는 YouTube oEmbed API |
| 노출 위치 | 메인화면 히어로 영역 (상단 가로 슬라이드) |

## 5. 레퍼럴 이벤트

### 5.1 구조

- A·B 모두 이벤트 자동 응모

### 5.2 이벤트 상세

| 항목 | 내용 |
|------|------|

| 보상 방식 | 추첨 (건수 비례 아닌 순수 추첨) |
| 보상 종류 | 상품권 (운영 정책으로 결정) |
| 중복 참여 | 추천 건수 누적 가능 (응모권 누적) |
| 금지 | 병원 방문·상담 완료 연동 트리거 |

## 6. 병원 탐색

### 6.1 구조 — A+B 듀얼

**진입 경로**

### 6.2 리스트 뷰

| 항목 | 내용 |
|------|------|

| 필터 | 시술 분야 (카테고리·세부 시술) / 지역 |
| 정렬 | 시술 분야 일치순 / 거리 가까운 순 / 태깅 후기 많은 순 |

**병원 카드 노출 항목**

| 항목 | 내용 |
|------|------|
| 병원명 + 대표 이미지 | — |
| 전문 시술 분야 | 칩 형태 최대 3개 |
| 위치·거리 | 예: "강남구 · 1.2km" |
| 태깅 후기 수 | 예: "후기 23개" |

### 6.3 지도 뷰

| 항목 | 내용 |
|------|------|

### 6.4 병원 상세 페이지

## 7. 1:1 채팅

### 7.1 기본 구조

| 항목 | 내용 |
|------|------|

### 7.2 메시지 유형

| 유형 | 내용 |
|------|------|
| 텍스트 | 최대 1,000자 |
| 사진 | 최대 5장/회, 10MB 이하 |
| 시스템 메시지 | 채팅 개설 안내 / 종료 안내 / 자동 종료 예고 알림 |

### 7.3 채팅 종료 조건 — 3가지 모두 적용

| 조건 | 내용 |
|------|------|
| 유저 직접 종료 | "상담 종료" 버튼 → 확인 다이얼로그 → 종료 |
| 상담 완료 처리 | 유저 또는 병원이 "상담 완료" → 양측 확인 후 종료 |
| 자동 종료 | 마지막 메시지 기준 30일 비활성 시 자동 종료 (운영 정책으로 조정 가능) |

종료 후 채팅방은 읽기 전용 전환 (재개설 불가, 기록만 열람)

### 7.4 채팅 기록 보관

| 항목 | 내용 |

## 11. 기술 스택

> 목표: **10만 동접자** 대응. 로드밸런서 기반 수평 확장, Terraform으로 인프라 코드 관리.

### 11.1 클라이언트

| 영역 | 기술 | 비고 |
|------|------|------|
| 모바일 앱 | Flutter (Dart) + **Riverpod** | iOS / Android 동시 대응. Riverpod으로 상태 관리 |
| 웹 CMS | Next.js (App Router) + TypeScript + **shadcn/ui** | 관리자 전용. shadcn/ui로 UI 컴포넌트 |

### 11.2 백엔드 코어

| 영역 | 기술 | 비고 |
|------|------|------|
| API 서버 | **Go** (Gin 또는 Echo 프레임워크) | 고성능 / 낮은 메모리 / 동시성 강점. 10만 동접에 적합 |
| 데이터베이스 | PostgreSQL | 관계형 데이터 (유저, 병원, 상담 요청 등) |
| 커넥션 풀러 | PgBouncer | 다중 인스턴스 환경에서 DB 커넥션 고갈 방지 |
| 캐시 + 세션 | **Redis** | DB 부하 분산, Centrifugo pub/sub 브로커 겸용, 세션 관리 |
| 파일 스토리지 | MinIO | S3 호환, 자체 호스팅. 비포&애프터 사진, 상담 사진 |
| 실시간 채팅 | Centrifugo + WebSocket | 자체 구축, Redis 브로커 연동. 100만 동접 검증 스택 |

### 11.3 인프라

| 영역 | 기술 | 비고 |
|------|------|------|
| 컨테이너화 | Docker | 전 서비스 컨테이너 운영 |
| 오케스트레이션 | Kubernetes (K8s) | Go API 서버 HPA 수평 확장 |
| **로드밸런서** | Nginx + K8s LoadBalancer | 10만 동접 트래픽 분산. Nginx L7 + K8s L4 이중 구성 |
| 인프라 코드 관리 | **Terraform** | 서버·네트워크·DB 등 인프라 전체 코드로 관리 (IaC) |
| CDN | Cloudflare | MinIO 앞단. 이미지·정적 파일 배포, 속도 개선 |

### 11.4 비동기 처리

| 영역 | 기술 | 비고 |
|------|------|------|
| 메시지 큐 | RabbitMQ | 상담 요청 발송, 푸시 알림, 이미지 리사이징 등 비동기 작업 |

### 11.5 검색

| 영역 | 기술 | 비고 |
|------|------|------|
| 병원·시술 검색 | Elasticsearch | 병원명·시술 키워드 전용. PostgreSQL 풀텍스트는 10만 동접 시 병목 가능 |

### 11.6 앱 기능

| 영역 | 기술 | 비고 |
|------|------|------|
| 지도 | 카카오맵 / 네이버맵 | **미정** |
| Flutter 지도 패키지 | `kakao_map_plugin` / `flutter_naver_map` | API 확정 후 선택 |
| 유튜브 썸네일 | `img.youtube.com` + oEmbed API | 앱 내 재생 없음 |
| 인증 | JWT + 카카오 소셜 로그인 | Apple 로그인 2차 추가 |
| 푸시 알림 | FCM (Firebase Cloud Messaging) | 상담 수신, 채팅, 자동 종료 예고 등 |
| 키워드 필터링 | 서버사이드 정규식 (Go) | 금액 기재 차단 |

### 11.7 모니터링 및 에러 트래킹

| 영역 | 기술 | 비고 |
|------|------|------|
| 인프라 모니터링 | Prometheus + Grafana | 동접자 수, API 응답시간, DB 커넥션 풀 실시간 모니터링 |
| 에러 트래킹 | Sentry | Flutter 앱 + Go 서버 에러 수집 |

### 11.8 아키텍처 흐름

```
Flutter 앱 (Riverpod)
    │
    ├─ REST API ──→ Nginx (L7) ──→ K8s LoadBalancer (L4)
    │                                     │
    │                              Go API 서버 (다중 인스턴스, K8s HPA)
    │                                     │
    │                                     ├──→ PostgreSQL (PgBouncer 경유)
    │                                     ├──→ Redis (캐시 + 세션)
    │                                     └──→ RabbitMQ ──→ Worker (푸시/이미지)
    │
    ├─ WebSocket ──→ Centrifugo ──→ Redis (pub/sub)
    │
    └─ 이미지 요청 ──→ Cloudflare CDN ──→ MinIO

인프라 전체: Terraform으로 코드 관리 (IaC)
CMS: Next.js + shadcn/ui
```

### 11.9 병목 포인트별 대응

| 병목 | 원인 | 대응 |
|------|------|------|
| API 서버 과부하 | 동접 급증 | Nginx + K8s LoadBalancer 이중 분산 + HPA 자동 확장 |
| DB 커넥션 고갈 | 다중 인스턴스 동시 접속 | PgBouncer 커넥션 풀링 |
| 이미지 로딩 느림 | 비포&애프터 사진 트래픽 집중 | Cloudflare CDN + MinIO 분리 |
| 검색 느림 | 병원 탐색 키워드 검색 | Elasticsearch 전용 인덱스 |
| 채팅 지연 | WebSocket 연결 폭증 | Centrifugo 클러스터링 + Redis 브로커 |
| 인프라 변경 리스크 | 수동 설정 오류 | Terraform IaC로 전체 인프라 코드화, 변경 이력 관리 |

## 13. MVP 범위 및 로드맵

### MVP-A (클로즈드 베타, 8–10주 목표)

- 역경매 상담 요청 플로우 (갤러리 업로드, 카메라 가이드 없이)
- 병원 상담 발송 + 금액 키워드 필터링 + 1:1 채팅 (Centrifugo)
- 비포&애프터 게시판 (자율 커뮤니티)
- 병원 탐색 (리스트 뷰 + 지도 뷰)
- 유튜브 히어로 (운영사 자체 채널)
- CMS: 병원 관리 + 신고 처리 + 유튜브 URL 관리

### MVP-B (정식 출시, +4–6주)

- 카메라 가이드 프레임 (부위별 실루엣 오버레이)
- 자유게시판 / Q&A
- 투표·설문 기능
- 레퍼럴 이벤트
- CMS 전체 기능 + 통계 대시보드
- Apple 로그인

### 3차 이후

- 수익 모델 확정 및 적용 (1안/2안 법률 검토 후)
- 커뮤니티 피드 알고리즘 고도화
- 비대면 화상 상담 내재화
- 병원 탐색 고도화 (정렬·필터 확장)
