## 3. 핵심 기능 비교

### 3-1. 매칭 시스템 (**가장 큰 차이**)

**A (ours) — 코디네이터 수동 매칭**
```
유저 → 상담 요청 등록 → 코디네이터가 검토 (SLA 2시간)
→ 2~3개 병원 수동 선택 → 채팅방 자동 생성 → 유저 비교 후 선택
```
- 사람이 병원을 선택 (품질 보장)
- CMS에 코디네이터 대시보드 구현
- 배민 스타일 4단계 진행 UI (접수→검토→선정→완료)
- 내부 매칭 노트 필드

**B (develop) — 자동 알고리즘 매칭**
```
유저 → 상담 요청 등록 → BullMQ 잡 실행
→ 점수 계산 (전문 40% + 거리 30% + 응답률 20% + 부하 10%)
→ 상위 3~5개 자동 배정 → 채팅방 생성
```
- 알고리즘이 병원 선택 (확장성)
- Haversine 거리 계산 (위도/경도)
- 2시간 내 매칭 실패 시 반경 2배 확대 후 재시도
- Redis 캐시로 응답률 시간당 갱신

**평가**: A가 비즈니스적으로 더 성숙 (코디네이터 = 품질 게이트키핑), B가 기술적으로 더 확장 가능 (자동화). 실제 서비스에서는 **A의 수동 매칭 + B의 점수 기반 추천을 코디네이터에게 제공**하는 하이브리드가 이상적.

## 9. 권장 사항

### 즉시 반영할 것 (B에서 가져올 아이디어)

1. **리뷰 시스템 추가** — B의 Review 모델 참고하여 구현 (매칭 기반 작성 자격, 별점, 사진)
2. **알림 매트릭스** — B의 14개 이벤트 기반 FCM 알림 설계 도입
3. **키워드 필터 정규식** — B의 구체적 패턴 (`\d{1,4}\s*(만\s*원|원|만)`, 전화번호, 카카오ID) 서버에 적용
4. **채팅 면책 고지** — 상단 고정 + dismiss 불가 처리
5. **에러 응답에 code 필드 추가** — `{ "error": "msg", "code": "CODE" }` 형태로 표준화
6. **API cursor 기반 페이징 고려** — 대규모 데이터셋에서 offset보다 효율적

### 유지할 것 (A의 장점)

1. **코디네이터 수동 매칭** — 의료 서비스 품질 보장의 핵심 차별점
2. **Go 서버** — 이미 전체 구현 완료, TypeScript 전환 비용 대비 이점 없음
3. **캐스트 멤버 시스템** — B에 없는 핵심 차별화 기능
4. **수익모델 설계** — 의료법 준수 구조가 잘 설계됨
5. **프리미엄/광고 시스템** — 이미 구현 완료

### 장기 검토

1. **Cloudflare R2 마이그레이션** — 프로덕션 배포 전 로컬 디스크 → R2 전환 필요
2. **Apple 로그인** — iOS 앱스토어 심사 필수 요건
3. **라이트 테마 지원** — 사용자 선호도에 따라 추후 추가
4. **Before/After 슬라이더** — 커뮤니티 UX 개선에 효과적

## 10. 결론

**A(ours)와 B(develop)는 같은 제품을 전혀 다른 관점에서 설계한 문서.**

- A는 **비즈니스 관점** (왜, 무엇을) — 수익모델, 법률, 캐스트 생태계
- B는 **구현 관점** (어떻게) — 화면 스펙, API 스펙, DB 스키마

기술 스택이 완전히 다르고 (Go vs TypeScript), 매칭 방식이 근본적으로 다르므로 (수동 vs 자동), **B의 코드를 그대로 병합하는 것은 불가능**. 다만 B의 **설계 아이디어** (리뷰, 알림, 키워드 필터 정규식, 방문 예약 카드, B/A 슬라이더 등)는 우리 Go 서버에 선별적으로 구현하는 것을 권장.
