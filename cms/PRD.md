# CMS Admin PRD v2.0
> 재사용 가능한 CMS 엔진 (코드 포크형) — 모듈 1~14 상세 설계
> v2.0: AI 자동 글쓰기 / 페르소나 / 외부 콘텐츠 소스 모듈 추가 반영

---

## 프로젝트 개요

**목적**
기술 블로그, 커뮤니티, 쇼핑몰, 뷰티 블로그 등 다양한 서비스에 포크해서 쓸 수 있는 범용 CMS 어드민 엔진 제작.

**구조**
멀티테넌트 아님. 단일 사이트 구조로 운영. 포크 시 별도 인스턴스로 분리.

**언어**
한국어 우선 구현. DB는 처음부터 다국어 대비 구조로 설계 (translations 테이블).
나중에 일본어/영어 추가 시 컬럼만 추가하면 됨. 테이블 구조 변경 불필요.

**게시물 언어 지원 (v2.0 추가)**
게시물 단위로 언어 설정 가능 (KO / JA). AI 자동 번역 연동.
어드민 UI는 한국어 고정. 게시물 콘텐츠만 다국어 관리.

**통계 정책 (GA4 전역 정책)**
아래 항목은 Google Analytics 4 연동으로 대체. 자체 구현하지 않음:
- 페이지별 조회수 / 유입 경로 분석 (UTM 추적) / 재방문율
- 평균 세션 시간 / 이탈률 / 코호트 분석
- 메뉴 접근 로그 / 전환율

**이유**: GA4가 이미 완벽하게 만들어져 있고 무료. 자체 구현하면 서버 비용 + 개발 공수 + 유지보수 부담만 생김.

## 모듈 1. 사이트 설정

서비스 전역에 영향을 주는 설정 모듈. 게시판이 0개여도 항상 존재.

### 1-1. 기본 정보

| 항목 | 사용처 |
|------|--------|
| 사이트명 | 브라우저 탭, 이메일 발신자명 |
| 사이트 URL | og태그, 카카오 공유, 카노니컬 URL |
| 대표 로고 | 헤더, 이메일 템플릿 |
| 파비콘 | 브라우저 탭 아이콘 |
| 대표 이메일 | 고객 문의 수신, 발신 이메일 |
| 사업자 정보 | 사업자번호, 대표자명, 주소 (푸터 하단) |
| 저작권 문구 | 푸터 카피라이트 |

### 1-2. SEO / AEO / GEO (전역 기본값)

게시물 단위에서 설정값이 없을 때 fallback으로 사용.

> **용어 통일**: AO → AEO (Answer Engine Optimization) 로 변경. AI 검색 인용 최적화.

**SEO 탭**
- 기본 타이틀 포맷: `{페이지명} | 사이트명`
- 기본 meta description (fallback)
- 기본 og:image (공유 썸네일 없을 때 fallback)
- GA4 측정 ID (`G-XXXXXXXXXX`)
- 네이버 웹마스터 인증 코드
- 구글 서치콘솔 인증 코드

**AEO 탭** (Answer Engine Optimization — AI 검색 인용 최적화)
- 사이트 핵심 소개문: AI가 이 사이트를 인용할 때 쓸 문구
- 핵심 FAQ 구조 등록 (GPT/Gemini/Perplexity 인용 대비)

**GEO 탭** (Generative Engine Optimization)
- 기본 구조화 데이터 타입: Organization / WebSite schema

### 1-3. 테마 설정 (v2.0 추가)

### 1-7. 다국어 구조 (translations 테이블)

```sql
CREATE TABLE translations (
  id         BIGINT PRIMARY KEY,
  key        VARCHAR(255) UNIQUE NOT NULL,  -- "menu.community"
  ko         TEXT NOT NULL,                 -- "커뮤니티" (현재 구현)
  ja         TEXT,                          -- NULL (나중에 추가)
  en         TEXT,                          -- NULL (나중에 추가)
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

적용 대상: 메뉴명, 게시판명, 카테고리명, 배너 텍스트, 팝업 내용, 이메일 템플릿, 알림 메시지, 시스템 메시지

### 2-4. AI API (v2.0 추가)

| 서비스 | 용도 | 키값 |
|--------|------|------|
| Google Gemini | 콘텐츠 생성 / 번역 / 점수 평가 | API Key |
| NanoBanana | 썸네일 이미지 AI 생성 | API Key |
| OpenAI (예비) | GPT 기반 콘텐츠 생성 | API Key |

**AI 연동 설계**
- 사용 모델: `gemini-2.0-flash` (기본), 어드민에서 모델 전환 가능
- API 호출 실패 시 재시도: 3회 (지수 백오프)
- 일일 API 사용량 모니터링 + 임계값 초과 시 알림
- 테스트 호출 버튼: 실제 콘텐츠 1건 생성해서 미리보기

**API Rate Limiting 정책**

| 항목 | 설명 |
|------|------|
| 일일 최대 호출 수 | 어드민 설정. Gemini 기본: 500회/일 (유료 플랜에 맞춰 조정) |
| 임계값 도달 시 | 배치 생성 즉시 중단 + 슬랙/이메일 알림 |
| 적용 대상 | Gemini (생성/번역/점수), NanoBanana (이미지) 각각 별도 설정 |
| 실행 전 체크 | Config 배치 시작 전 잔여 쿼터 확인 → 부족 시 실행 취소 |

> 주의: Gemini 무료 티어는 분당 15회 제한. generateCount=30 × 3회 재시도 = 최대 90회 호출. 유료 전환 또는 일일 한도를 실제 플랜에 맞게 설정 필요.

**결제/환불 프로세스**

결제 연동 시 아래 흐름을 반드시 구현.

```
환불 신청 (유저)
  → 환불 상태: REQUESTED
  → 어드민 검토 → APPROVED / REJECTED
  → PG사 환불 API 호출 → PROCESSING
  → PG사 웹훅으로 완료 수신 → COMPLETED
  → 유저 이메일/앱 알림 발송
```

| 환불 정책 항목 | 내용 |
|-------------|------|
| 무조건 환불 기간 | 구매 후 7일 이내 (전자상거래법) |
| 부분 환불 계산 | 원가 - 수수료 - 사용료 |
| 거래명세서 | 환불 완료 시 자동 발급 |
| 거절 사유 | 상품 훼손, 약관 위배, 이용 완료 — 약관에 명시 필수 |

## 모듈 3. 회원 / 등급 / 권한

### 3-1. 등급 구조

| 등급 | 이름 | 기본 승급 조건 |
|------|------|---------------|
| 0 | 비회원 | — |
| 1 | 새싹 | 가입 즉시 |
| 2 | 일반 | 게시물 10개 + 댓글 20개 + 가입 7일 |
| 3 | 우수 | 게시물 50개 + 댓글 100개 + 가입 30일 |
| 4 | VIP | 게시물 100개 + 댓글 200개 + 가입 90일 |
| 5 | VVIP | 게시물 300개 + 댓글 500개 + 가입 180일 |
| 9 | 어드민 | 수동 지정만 가능 |

기본값은 어드민에서 자유롭게 수정 가능.

**승급 조건 설정 항목**
- 활동: 게시물 수, 댓글 수, 좋아요 받은 수, 출석 일수, 포인트 보유량
- 기간: 가입 경과일, 마지막 로그인
- 인증: 이메일 인증 완료, 본인인증 완료, 프로필 완성도
- 조건 방식: AND / OR 선택
- 자동 승급 ON/OFF, 승급 알림 ON/OFF

### 3-1-1. 개인정보 보존 및 삭제 정책

개인정보보호법 준수를 위한 데이터 보존 기간 및 삭제 흐름 정의.

| 데이터 종류 | 보존 기간 | 근거 |
|-----------|---------|------|
| 회원 기본정보 (이메일/이름) | 탈퇴 후 90일 유예 → 완전 삭제 | 분쟁 대비 |
| 결제 정보 | 거래일로부터 5년 | 전자상거래법 |
| 어드민 활동 로그 | 1년 | 법적 감사 대응 |
| 게시물 수정 이력 | 90일 | 분쟁 대비 |
| 로그인 이력 | 1년 | 보안 감사 |
| 알림 발송 이력 | 6개월 | 자체 정책 |

**탈퇴 처리 흐름**
```
탈퇴 신청
  → 즉시: 로그인 차단, 세션 무효화
  → 즉시: 이메일/전화번호 마스킹 (게시물/댓글 익명화, 삭제 아님)
  → 90일 경과 후: 개인식별 정보 완전 삭제
  → 결제 정보는 5년 보존 후 삭제
```

**참고**: 일본 서비스 시 APPI(일본 개인정보보호법) 별도 검토 필요.

### 3-2. 어뷰징 방지 시스템 (6 레이어)

등급 조건 달성을 위한 도배/자동화 방지. 레이어를 겹쳐서 방어.

**레이어 1 — 품질 필터**
- 게시물 최소 100자, 댓글 최소 20자 (어드민 설정)
- 동일 문구 반복 감지 → 카운트 제외

**레이어 2 — 체류 시간**
- 게시물 작성: 에디터 열고 최소 30초 후 발행 (어드민 10~300초)
- 댓글 작성: 최소 10초

**레이어 3 — 일일 상한**
- 하루 인정 게시물 최대 5개, 댓글 최대 10개
- 연속 작성 간격 3분 이내 제외

**레이어 4 — 신뢰도 기반**
- 좋아요 / 게시물 비율 이상 감지
- 신고 당한 횟수 반영
- 블라인드 처리된 게시물 카운트 차감

**레이어 5 — 디바이스 핑거프린팅**
- FingerprintJS 사용
- 브라우저 + OS + 해상도 + 폰트 + 타임존 + GPU 조합으로 고유 ID 생성
- 다중 계정 감지 → 자동 승급 보류

**레이어 6 — 좋아요 품앗이 방지**
- 1인 1회 토글, 취소 후 재좋아요 불가
- 하루 좋아요 발신 기본 50개 제한
- 맞좋아요 패턴 감지 → 자동 승급 보류

### 3-5. 휴면 계정 처리

```
90일 미접속
  → 휴면 예정 알림 발송 (이메일 + 카카오)
알림 후 7일 무반응
  → 휴면 전환 (쓰기/댓글 권한 제한)
로그인 시도
  → 본인인증
  → 즉시 정상 전환
```

휴면 기간 어드민 설정 가능 (기본값 90일)

## 모듈 4. 게시판 관리

### 4-1. 계층 구조

```
board_groups (대분류)
  └─ boards (게시판, 실제 운영 단위)
       └─ board_categories (소분류)
            └─ posts (게시물)
                 └─ comments (댓글/대댓글)
```

### 4-2. 게시판 타입 7종

| 타입 | 설명 | 주요 특징 |
|------|------|----------|
| 일반 | 제목 + 본문 | 기본형 |
| 갤러리 | 이미지 중심 | 그리드뷰, 대표 이미지 표시 |
| 자료실 | 파일 첨부 중심 | 다운로드 수 집계 |
| QA | 질문 + 답변 | 채택 기능, 미해결/해결 필터 |
| 동영상 | 영상 임베드 | 유튜브/Vimeo 썸네일 자동 추출 |
| 일정/캘린더 | 날짜 기반 | 월별/주별 보기 |
| 투표 | 투표 + 결과 | 복수 선택, 결과 시각화 |

### 4-3. 슬러그 구조

```sql
CREATE TABLE boards (
  id          BIGINT PRIMARY KEY,
  parent_id   BIGINT,               -- null이면 최상위 대분류
  slug        VARCHAR(100) NOT NULL, -- 개별 슬러그 (예: free)
  full_path   VARCHAR(500) NOT NULL, -- 전체 경로 (예: /community/free)
  depth       TINYINT NOT NULL,      -- 0: 대분류, 1: 게시판, 2: 소분류
  ...
);
```

**슬러그 규칙**
- 한글 입력 시 영문 자동 변환 제안
- 공백 → 하이픈 변환, 특수문자 제거, 대문자 → 소문자
- 중복 불가, 변경 시 301 리다이렉트 자동 등록

### 4-4. 게시판 생성 설정값

**기본 설정**
- 게시판명, 슬러그, 타입, 설명, 썸네일, 정렬 순서, 공개 여부

**권한 설정** (등급 기준, 0=비회원 ~ 9=어드민만)
- 읽기 / 쓰기 / 댓글 / 파일 업로드 / 좋아요

**게시물 설정**
- 공지 고정 허용, 익명 작성, 비밀글, 첨부파일 허용
- 글자 수 제한 (최소/최대), 예약 발행, 게시물 이동 권한
- 신고 임계값 (게시판별 자동 블라인드 기준)

**인터랙션 설정**
- 좋아요 / 싫어요 / 댓글 / 대댓글 / 공유 / 조회수 각각 ON/OFF

**스킨 설정**
- 리스트형 / 카드형 / 앨범형
- 페이지당 게시물 수

### 4-5. 게시물 이력 관리

분쟁 발생 시 원본 추적 목적.

- 수정 버전별 기록 (post_revisions 테이블)
- 어드민 / 작성자 본인만 이력 열람
- 이전 버전으로 복원 가능
- 게시물에 "수정됨" 표시 (최종 수정 시각 포함)
- 보존 기간: 기본값 90일 (어드민 설정)

### 4-6. 통계

- **GA4 대체**: 게시판별 페이지 조회수, 유입 경로
- **자체 구현**: 게시물 수, 댓글 수, 좋아요 수, 신고 수, 인기 게시물 TOP 10

### 5-1-1. AI 생성 콘텐츠 표기 정책 (v2.0 추가)

`aiGenerated: true`인 게시물은 사용자 화면과 시스템 메타데이터에 반드시 표시.

| 표시 위치 | 방식 |
|----------|------|
| 게시물 상단 | "AI 보조 생성 콘텐츠" 안내 배너 (어드민 설정으로 ON/OFF) |
| 어드민 목록 | `AI` 뱃지 표시 |
| JSON-LD | `"author": {"@type": "Person", "name": "페르소나명 (AI)"}` |
| 이용약관 | "본 사이트는 AI 보조로 생성된 콘텐츠를 포함합니다" 조항 필수 |

**이유**: AI 생성 콘텐츠를 사람이 쓴 것처럼 오인하게 하면 허위 광고 위험. SEO 측면에서도 Google이 AI 스팸 콘텐츠에 패널티를 부과하므로 투명성 확보가 장기 검색 순위에 유리.

| 점수 | 설명 | 평가 기준 |
|------|------|----------|
| SEO | 검색 엔진 최적화 | 키워드 밀도, 메타 태그, 제목 구조, 내부 링크 |
| AEO | AI 검색 인용 최적화 | 질문-답변 구조, 명확한 정의, FAQ 포함 여부 |
| GEO | 생성형 엔진 최적화 | 구조화 데이터, 팩트 밀도, 인용 가능성 |

- 점수 범위: 0~100
- 어드민 게시물 목록에 3가지 점수 컬럼 표시 + 점수 기준 필터링

- 사용자 작성 글: 제출 시 백그라운드 자동 채점 → 어드민만 확인 가능
- 점수 낮은 게시물 목록 → 어드민이 수동 보완 또는 블라인드 처리

### 5-5. 이미지 업로드 정책

**결정: 용량/개수 제한 없음**

이유: 과도한 제한은 UX를 해침. 디시인사이드처럼 자유롭게 업로드 허용.
스토리지 비용은 Cloudflare R2로 최소화 (이그레스 무료).

| 항목 | 정책 |
|------|------|
| 용량 제한 | 없음 |
| 개수 제한 | 없음 |
| 허용 확장자 | jpg, jpeg, png, gif, webp, heic |
| 차단 확장자 | exe, php, js, sh, bat, cmd 등 실행파일 |
| MIME 타입 검사 | 확장자 위조 감지 (서버에서 2차 검사) |
| 악성코드 감지 | 이미지 내 스크립트 삽입 패턴 감지 |
| 실패 메시지 | "허용되지 않는 파일입니다" |

**이미지 최적화 자동 처리** (업로드 즉시)
```
업로드 파일 수신
  → EXIF 메타데이터 제거 (GPS, 기기 정보, 촬영 시각)
  → 리사이즈 (최대 1920px 초과 시 자동 축소)
  → WebP 변환 + 압축 (품질 85%)
  → 썸네일 자동 생성 (400x400px, center crop)
  → PostgreSQL BYTEA 컬럼에 바이너리로 저장
```

**이미지 저장 방식 (현재)**

결정: PostgreSQL에 WebP 바이너리로 직접 저장.
이유: R2/S3 연동 설정 없이 빠르게 시작 가능. 나중에 스토리지 레이어만 교체하면 됨.

```sql
-- media 테이블 핵심 컬럼
id           BIGINT PRIMARY KEY
user_id      BIGINT
file_name    VARCHAR(255)       -- 원본 파일명
mime_type    VARCHAR(50)        -- "image/webp"
size_bytes   INTEGER            -- 변환 후 용량
width        INTEGER
height       INTEGER
data         BYTEA              -- WebP 바이너리 데이터
thumb_data   BYTEA              -- 썸네일 바이너리 (400x400)
created_at   TIMESTAMP
```

**BYTEA 저장 한계 및 마이그레이션 기준**

| 항목 | 설명 |
|------|------|
| 저장 범위 | BYTEA는 **썸네일(400x400px)만** 저장. 원본 이미지는 로컬 파일 경로 |
| 마이그레이션 기준 | 이미지 총량 **1GB 초과** 시 R2/S3 전환 착수 |
| 이유 | PostgreSQL BYTEA는 WAL 로그에 전체 바이너리 포함 → 디스크/백업 비용 급증 |

**나중에 R2/S3 마이그레이션 시**
- BYTEA → R2 업로드 후 URL 저장으로 교체
- 비즈니스 로직 변경 없음, 저장 레이어만 교체
- media 테이블에 `url VARCHAR` 컬럼 추가 후 data/thumb_data 컬럼 제거

## 모듈 6. 신고 / 제재 관리

### 6-1. 신고 구조

**신고 대상**: 게시물, 댓글, 유저
**신고 사유**: 스팸/도배, 욕설/비방, 음란물, 개인정보 노출, 저작권 침해, 불법 정보, 기타(직접 입력 최대 200자)

### 6-2. 자동 처리 흐름

```
신고 접수
  → 신고 어뷰징 감지 (집단 신고 가중치 조정)
  → 누적 카운트 증가
  → 임계값 미달: 어드민 대기 목록에 추가
  → 임계값 도달: 자동 블라인드 + 어드민 알림 발송

어드민 처리
  → 블라인드 / 삭제 / 복구 / 반려
  → 유저 제재 (경고/정지/영구)
  → 제재 이력 저장
  → 경고 누적 → 자동 정지
```

임계값은 게시판별로 다르게 설정 가능.

### 6-3. 유저 제재

| 제재 종류 | 설명 |
|----------|------|
| 경고 | 누적 카운트 +1, 자동 정지 기준에 반영 |
| 일시 정지 | n일 기간 설정, 로그인 차단 |
| 영구 정지 | 로그인 영구 불가 |
| 강제 로그아웃 | 현재 세션 즉시 종료 |
| IP 차단 | 해당 IP 접근 완전 차단 |

**경고 누적 자동 처리** (어드민 설정 가능)
- 3회 → 7일 자동 정지
- 5회 → 30일 자동 정지
- 10회 → 영구 정지

### 6-4. 금칙어 시스템

**감지 패턴 3종**

| 유형 | 예시 | 감지 방법 |
|------|------|----------|
| 직접 욕설 | 씨발, 개새끼, 병신 | 금칙어 목록 직접 등록 |
| 변형 욕설 | 씨1발, s.i.b, 씌발 | 정규식 패턴으로 감지 |
| 초성 욕설 | ㅅㅂ, ㅂㅅ, ㄱㅅㄲ | 초성 패턴 목록 별도 관리 |

**어드민 관리 기능**: 기본 제공 목록 + 커스텀 추가/삭제, 패턴 테스트, 게시판별 금칙어 별도 설정

### 6-5. 욕설 감지 수위 3단계

**결정: 규칙 기반으로 시작**

이유: AI 감지는 게시물마다 API 호출 → 비용 예측 불가. 규칙 기반으로 먼저 운영하고 나중에 AI 레이어 추가.

| 수위 | 직접 욕설 | 변형/초성 | 음담패설 | 적용 예 |
|------|----------|----------|---------|---------|
| 🟢 허용 | ** 치환 | 통과 | 성인 태그만 | VIP 게시판 |
| 🟡 보통 (기본값) | ** 치환 | ** 치환 | 게시 차단 | 자유게시판 |
| 🔴 엄격 | 게시 차단 | 게시 차단 | 게시 차단 | 공지, QA |

나중에 AI 업그레이드 시 모듈화 구조:
```
현재: 규칙 기반 감지 → 처리
나중: 규칙 기반 1차 → AI 2차 감지 → 처리
```

### 6-6. 신고 어뷰징 방지

특정 유저를 집단으로 신고 폭격하는 케이스 방어.

| 패턴 | 감지 기준 | 처리 |
|------|----------|------|
| 집단 신고 | 같은 IP에서 단시간 집중 신고 | 가중치 낮춤 |
| 대량 신고 | 1인이 단시간 n건 이상 신고 | 가중치 낮춤 |
| 반복 신고 | 같은 게시물 반복 신고 | 1회만 카운트 |
| 신고 오용 | 신고 반려 누적 n회 | 신고 권한 제한 |

### 7-2. 배너 관리

**배너 위치 5종**

| 위치 | 설명 |
|------|------|
| 메인 히어로 | 최상단 풀스크린 슬라이더 (속도/전환 효과 설정) |
| 서브 배너 | 게시판 상단 |

**배너 설정**
- 이미지 타입: PC / 모바일 / 태블릿 각각 등록
- 텍스트 타입: 텍스트 + 배경색/글자색 + 링크

- 노출 대상: 전체 / 비로그인만 / 로그인만
- A/B 테스트: 노출 비율 설정, CTR 비교, 성과 높은 쪽 수동/자동 선택
- 배너 그룹핑: 이벤트 단위로 묶어 한번에 ON/OFF
- 배너 라이브러리: 종료된 배너 보관 + 성과 데이터 함께 보관 + 재사용
- 배너 템플릿: 자주 쓰는 디자인 저장
- UTM 자동 생성: `utm_source=banner&utm_medium={위치}&utm_campaign={캠페인명}`

**닫기 옵션 처리 방식**

결정: 비로그인은 쿠키, 로그인은 DB 저장 (모든 기기 동기화)
이유: 로그인 유저가 A 기기에서 닫으면 B 기기에서도 안 뜨는 게 맞는 UX.

```
비로그인 → 쿠키 (만료일 = 닫기 옵션에 따라 자정/7일/30일/1년)
로그인   → banner_dismissals 테이블에 저장
           (user_id, banner_id, dismiss_type, expires_at)
```

어드민에서 게시판별로 제공할 닫기 옵션 선택 가능:
- 그냥 닫기 / 오늘 하루 / 1주일 / 한달 / 영구

**통계**
- 자체 구현: 클릭수, 노출수, CTR, 디바이스별 클릭률
- GA4 대체: 유입 경로 분석, 전환율

### 7-3. 팝업 관리

- 타입: 이미지 팝업 / HTML 팝업 (직접 HTML 입력)
- 노출 위치: 전체 / 메인 / 특정 게시판
- 크기/위치: 가로/세로 px, 화면 내 X/Y 좌표
- 닫기 옵션: 배너와 동일한 방식 (쿠키 + DB 병행)

## 모듈 8. 알림 관리

### 8-0. 알림 큐 구현 방향

**구현 방식**: NotificationQueue 테이블 폴링 (초기 — Redis 없이 시작)

| 우선순위 | 종류 | 처리 방식 |
|---------|------|----------|
| 1 (긴급) | 인증 메일, 계정 정지, 보안 알림 | 즉시 발송 (동기 API 호출) |
| 2 (일반) | 댓글, 좋아요, 등급 변경 | 1분 배치 (cron 폴링) |
| 3 (저우선) | 마케팅, 뉴스레터 | 야간 배치 (22시 이후 처리) |

```
NotificationQueue 테이블 폴링 흐름:
  → 긴급(priority=1): 이벤트 발생 즉시 동기 발송
  → 일반/저우선: 큐 테이블에 INSERT → cron이 주기적으로 처리
  → 실패 시 retryCount 증가 → 최대 3회 후 failed 처리 + 어드민 알림
```

**추후 확장**: 트래픽 증가 시 BullMQ(Redis) 전환. 큐 로직이 NotificationQueue 테이블 기반이므로 비즈니스 로직 변경 없이 저장소만 교체 가능.

### 8-1. 알림 채널 5종

| 채널 | 구현 방식 |
|------|----------|
| 푸시 알림 (PWA) | 서비스 워커 + 브라우저 푸시 API |
| 이메일 | SMTP / SendGrid (API 통합 설정 연동) |
| 카카오 알림톡 | 카카오 비즈니스 채널 (템플릿 사전 승인 필요) |
| SMS / LMS | 솔라피 / NHN (API 통합 설정 연동) |
| 인앱 알림 | 헤더 알림 벨, 드롭다운 목록 |

### 8-2. 알림 우선순위

**결정: 3단계로 분리**

이유: 우선순위 없으면 대량 마케팅 메일 처리 중 인증 메일이 밀려 늦게 가는 문제 발생.

| 우선순위 | 종류 | 처리 방식 |
|---------|------|----------|
| 긴급 | 계정 정지, 보안 알림, 인증 메일, 비밀번호 재설정 | 즉시 발송 |
| 일반 | 댓글, 좋아요, 등급 변경, 포인트 변경 | 순차 발송 |
| 저우선 | 마케팅, 이벤트, 뉴스레터 | 여유 있을 때 발송 |

### 8-3. 알림 큐 시스템

대량 발송 시 서버 과부하 방지. n건씩 순차 처리.
우선순위 높은 큐가 먼저 처리됨.

### 8-4. 알림 재발송

발송 실패 시 자동 재시도.
- 재시도 횟수: 기본값 3회 (어드민 설정)
- 재시도 간격: 1분 / 5분 / 30분
- 최종 실패 시: 어드민 알림 + 실패 로그 기록

### 8-5. 알림 그룹핑

같은 게시물에 댓글 10개 달리면 알림 10개 아니라 묶어서 1개 발송.

```
"홍길동 외 9명이 내 게시물에 댓글을 달았습니다"
```

그룹핑 기준: 기본값 1시간 내 동일 타입 묶음 (어드민 설정)

### 8-6. 알림 트리거 목록

| 분류 | 트리거 |
|------|--------|
| 게시물 | 댓글 달림, 대댓글 달림, 좋아요 n개 달성, 인기글 선정, @ 멘션 |
| 회원 | 가입 환영, 등급 승급, 포인트 변경, 경고, 정지, 휴면 예정 |

| 어드민 수동 | 전체 공지, 특정 등급 대상, 개별 유저 |
| AI 자동 글쓰기 (v2.0) | 자동 글쓰기 실행 완료, 실패 알림, 일일 한도 도달 |

### 8-7. 이메일 관리

**기본 제공 템플릿**
- 회원가입 환영 / 비밀번호 재설정 / 이메일 인증
- 등급 승급 축하 / 계정 경고 / 계정 정지 / 휴면 예정

**편집 기능**
- HTML 에디터
- 변수 치환: `{{이름}}`, `{{등급}}`, `{{포인트}}`, `{{링크}}`
- 미리보기 (실제 발송 모습 확인)
- 테스트 발송 (내 이메일로 확인)

**수신 거부 (법적 필수)**
마케팅 메일은 정보통신망법상 수신 거부 링크 필수.
- 이메일 하단에 "수신 거부" 링크 자동 삽입
- 클릭 시 자동으로 email_unsubscribes 테이블에 기록
- 수신 거부된 유저에게는 마케팅 메일 발송 안 됨 (시스템 메일은 계속 발송)

### 8-8. 알림 구독 관리

**유저가 설정 가능한 것**
- 채널별 ON/OFF (인앱/이메일/카카오/SMS 각각)
- 게시판별 구독: "자유게시판 새 글 알림만 받기"
- 특정 유저 구독: "이 유저 글 올리면 알림"
- 키워드 알림: "이 키워드 포함된 글 올리면 알림"

**어드민이 설정하는 것**
- 필수 알림 지정 (유저가 끌 수 없음): 계정 정지, 경고
- 야간 발송 제한: 22시~08시 차단 (어드민 설정)
- 일일 최대 발송 제한 (유저당 n건)

### 8-9. 발송 시간대 최적화

현재는 구조만 잡아둠. 나중에 AI 연동 시 유저별 최적 시간대 자동 발송.
(이 유저는 오전 9시 이메일 열람률 높음 등)

### 8-10. 알림 통계

- 채널별 발송 수, 열람률, 클릭률
- 수신 거부율, 발송 실패율

슬랙, 잔디, 디스코드 연동에 활용.

## 모듈 10. 시스템 / 어드민 로그

### 10-1. 역할 기반 권한 (Role)

**결정: 3단계 고정 역할 → Role 직접 생성 방식으로 변경**

이유: 슈퍼/운영자/에디터 3단계로는 실제 운영 케이스를 못 커버함.
예) A는 자유게시판만, B는 회원 조회만, C는 통계만.

**구조**
```
Role 직접 생성
  → 모듈별 권한 설정 (조회/수정/삭제 각각 ON/OFF)
  → 게시판별 권한 세분화 (게시판 관리 ON일 때만 활성화)
  → 어드민 계정에 역할 할당
```

**기본 제공 Role** (수정은 가능, 삭제는 슈퍼 어드민만)
- 슈퍼 어드민: 전체 권한
- 운영자: 회원/게시물/신고 관리
- 에디터: 콘텐츠만

### 10-2. 어드민 활동 로그

모든 어드민 액션을 전부 기록.

**기록 항목**: 어드민 계정, 액션 유형, 변경 전 값(before), 변경 후 값(after), 대상, IP 주소, 일시

**로그 무결성**: 수정/삭제 불가. 슈퍼 어드민도 지울 수 없음.
**보존 기간**: 1년 (법적 근거 대응 목적)

### 10-3. 로그 복합 검색

어드민/액션/대상/기간/IP 조건 AND 조합 검색.
자주 쓰는 검색 저장 (예: "API 키 변경 모니터링").
변경 전/후 diff 뷰 (나란히 비교).

### 10-4. 보안

- **슈퍼 어드민 2FA 필수**: 선택 아님. 2FA 없이 로그인 불가.
- **세션 타임아웃**: 기본값 30분 비활동 시 자동 로그아웃
- **로그인 이력**: 언제/어느 IP에서 접근. 새 IP 접근 시 이메일 알림.
- **IP 허용 목록**: 특정 IP에서만 어드민 접근 가능하도록 설정
- **JWT 기반 인증 (v2.0)**: httpOnly 쿠키 + 7일 만료 + 미들웨어 보호

### 10-5. 작업 승인 시스템

민감한 작업은 혼자 처리 불가. 슈퍼 어드민 승인 필요.

- 회원 영구 정지
- 대량 게시물 삭제
- API 키 변경

### 10-6. 시스템 로그

- 에러 로그 (500 에러 발생 기록)
- 느린 쿼리 로그 (n초 이상 걸린 DB 쿼리)
- 외부 API 호출 실패
- 악성 파일 업로드 시도
- AI API 호출 실패 및 사용량 초과 (v2.0)

이상 징후 시 슬랙/이메일 자동 알림.

### 10-7. 기타

- 로그 내보내기 (CSV/엑셀, 법적 감사 대응용)
- 어드민별 알림 설정 (슈퍼=전체 / 운영자=신고/제재만 등)

## 모듈 13. AI 자동 글쓰기 (v2.0 신규)

외부 커뮤니티 트렌드를 기반으로 AI가 자동으로 블로그 포스트를 생성하는 모듈.
목표: Google + Perplexity 등 AI 검색 노출. 초반에는 AI로 기반을 쌓고, 트래픽이 늘면 AI 비중을 줄여나가는 전략.

### 13-1. 전체 파이프라인 (2단계)

```
[1단계: 배치 생성] generateHour 시각에 자동 실행
  외부 소스 수집 (모듈 14)
  → generateCount 만큼 배치 생성 (최대 100개)
  → 각 글: AI 콘텐츠 생성 (Gemini) + SEO/AEO/GEO 점수 평가
  → 임계값 통과? → draft 저장 + publishWindow 내 랜덤 scheduledAt 배정
  → 미통과? → 재생성 (최대 3회)
  → 3회 모두 실패? → 소스 isProcessed=true + SKIPPED 로그 (점수/피드백 포함)
  → (선택) 썸네일 이미지 생성 (NanoBanana)
  → (선택) 일본어 번역

[2단계: 자동 발행] publishWindowStart ~ publishWindowEnd 사이
  → GitHub Actions 5분마다 체크
  → scheduledAt ≤ 현재 시각인 draft 게시물 → published로 변경
  → 발행 완료 → SUCCESS 로그
```

**발행 시간 랜덤 배정 예시**
```
generateCount=30, publishWindowStart=09:00, publishWindowEnd=22:00 설정 시:

07:00 → 30개 배치 생성 (draft)
        각 글에 09:00~22:00 사이 랜덤 시각 배정
        예: 09:14, 09:47, 10:23, 11:08, ... 21:44

이후 5분마다 → scheduledAt 지난 draft 자동 발행
```

### 13-2. 자동 글쓰기 Config

한 개의 Config = 한 개의 자동 글쓰기 규칙.

| 설정 항목 | 설명 | 기본값 |
|----------|------|--------|
| 이름 | Config 식별명 | — |
| 활성 여부 | ON/OFF | OFF |
| 페르소나 | 글 쓸 페르소나 선택 | — |
| 게시판 | 발행할 게시판 선택 | — |
| 포스트 타입 | TREND / EVERGREEN | TREND |
| 언어 | KO / JA | KO |
| **글 생성 시각** | 배치 생성 시작 시간 (0~23시, KST 기준) | 07:00 |
| **일일 생성 수** | 하루 최대 배치 생성 개수 (최대 100) | 30 |
| **발행 시작 시각** | 랜덤 발행 윈도우 시작 (0~23시) | 09:00 |
| **발행 종료 시각** | 랜덤 발행 윈도우 종료 (0~23시) | 22:00 |
| 최소 SEO 점수 | 이 점수 미만이면 재생성 | 70 |
| 최소 AEO 점수 | 이 점수 미만이면 재생성 | 70 |
| 최소 GEO 점수 | 이 점수 미만이면 재생성 | 70 |
| **검수 필요** | ON 시 draft 상태에서 자동 발행 안 함, 어드민 수동 승인 필요 | OFF |
| 이미지 생성 | NanoBanana로 썸네일 자동 생성 | OFF |
| 자동 번역 | 한국어 원문 → 일본어 자동 번역 | OFF |

### 13-2-1. 검수 워크플로우 (검수 필요 ON 시)

```
AI 배치 생성 완료 → draft 저장 (자동 발행 보류)
  → 어드민 "검수 대기" 목록에 표시
  → 어드민 액션:
      [미리보기]  게시물 전체 내용 확인
      [수정 후 저장]  에디터에서 내용 수정 → draft 재저장
      [발행]  즉시 published 상태로 전환
      [거절]  draft 삭제 + 해당 소스 isProcessed=true
```

**검수 필요 OFF (기본값)**: 기존대로 scheduledAt 도달 시 자동 발행.

**일일 생성 수 가이드라인**
- 추천: 30개/일 (Google + Perplexity 안전선)
- 최대: 100개/일 (컨텐츠팜 패널티 주의)
- 외부 소스(성예사/걸스채널/코스메)에서 매일 수백 개 수집 가능

### 13-3. 포스트 생성 방식

**TREND 포스트**
- 수집된 외부 소스에서 최신 트렌드 분석
- "지금 화제인" 스타일, 시의성 강조
- 소스별 트렌드 키워드 추출 → Gemini에 주입

**EVERGREEN 포스트**
- 시간이 지나도 유효한 정보성 콘텐츠
- "완전 가이드", "처음부터 끝까지" 스타일
- 롱테일 키워드 타겟팅, 구조화된 답변 형태
- AEO/GEO 최적화: 질문-답변 구조, 팩트 밀도 높게

### 13-4. 점수 평가 및 재시도

재생성 루프 안에서 생성과 점수 평가를 함께 처리. 임계값 통과 시에만 draft 저장.

```
while (retryCount < 3):
  글 생성 (Gemini)
  → 점수 평가 (SEO/AEO/GEO)
  → 전체 임계값 통과? → break (draft 저장)
  → 미통과? → retryCount++ → continue (재생성)

3회 모두 실패:
  → 소스 isProcessed=true (무한 재시도 방지)
  → SKIPPED 로그 (점수 + Gemini feedback 기록)
```

- 점수 미달로 SKIPPED된 소스는 처리완료 마킹 (같은 소스 무한 반복 방지)
- 새 소스는 매일 외부 수집으로 보충

### 13-5. 자동 글쓰기 로그

| 필드 | 설명 |
|------|------|
| Config | 어떤 Config로 실행됐는지 |
| 상태 | SUCCESS / FAILED / SKIPPED |
| 포스트 타입 | TREND / EVERGREEN |
| 언어 | KO / JA |
| SEO 점수 | 생성된 글의 SEO 점수 |
| AEO 점수 | 생성된 글의 AEO 점수 |
| GEO 점수 | 생성된 글의 GEO 점수 |
| 재시도 횟수 | 0~3 |
| 에러 메시지 | 실패/SKIPPED 시 원인 + Gemini feedback |
| 실행 시각 | — |

### 13-6. 수동 실행

- 어드민이 직접 "지금 실행" 버튼으로 즉시 배치 생성 가능
- 실행 결과 즉시 확인 (로그 실시간 표시)
- 스케줄 외 추가 실행 (일일 한도와 별도)

### 13-7. Config 생성 UI (스텝바이스텝 Wizard)

```
Step 1: 기본 설정 (이름, 포스트 타입, 언어)
Step 2: 콘텐츠 소스 선택 (어느 외부 소스에서 가져올지)
Step 3: 페르소나 & 게시판 선택
Step 4: 품질 기준 (SEO/AEO/GEO 임계값)
Step 5: 스케줄 설정
  - 글 생성 시각 (generateHour, 기본: 07시)
  - 일일 생성 수 (최대 100, 기본: 30)
  - 발행 윈도우 시작/종료 (기본: 09시~22시)
Step 6: 이미지/번역 옵션
Step 7: 확인 & 저장
```

### 13-8. Cron 스케줄러

GitHub Actions Scheduled Workflow 방식. 서버 crontab 불필요.

**타임존 정책**
- `generateHour`, `publishWindowStart`, `publishWindowEnd` 입력값은 모두 **Asia/Seoul (KST) 기준**
- GitHub Actions schedule은 UTC 기준이므로 KST-9 변환 적용 (예: KST 07:00 → UTC 22:00 전날)
- 발행 시각 편차: ±5분 허용 (배치 생성 특성상 실용적으로 무관)

```yaml
# 배치 생성: 매시간 실행 → generateHour 일치하는 config만 처리
schedule: "0 * * * *"
→ POST /api/v1/cron/auto-posting (X-Cron-Secret 인증)

# 자동 발행: 5분마다 실행 → scheduledAt 지난 draft 발행
schedule: "*/5 * * * *"
→ POST /api/v1/cron/publish (X-Cron-Secret 인증)
```

**장점**: 서버 직접 접근 불필요, GitHub UI에서 실행 내역/실패 알림 확인, 수동 트리거 가능

## 모듈 14. 외부 콘텐츠 소스 (v2.0 신규)

AI 자동 글쓰기의 원료가 되는 외부 커뮤니티 데이터 수집 모듈.

### 14-1. 지원 소스

| 소스 | 설명 | 언어 | 특징 |
|------|------|------|------|
| 성예사 (Sungyesa) | 한국 뷰티 커뮤니티 | KO | 성분 분석, 뷰티 리뷰 |
| 걸스채널 (Girlschannel) | 일본 여성 커뮤니티 | JA | 일본 트렌드, 여성 관심사 |
| 코스메 (Cosme) | 일본 화장품 리뷰 사이트 | JA | 화장품 상세 리뷰, 평점 |

### 14-2. 수집 방식

```
웹 스크래핑 (HTML 파싱)
  → 각 사이트 페이지 크롤링 (Cheerio / Puppeteer 등)
  → 최신 게시물/리뷰 주기적 수집
  → ContentSource 테이블에 저장
  → 중복 방지 (externalId 기준)
  → isProcessed: false (미처리 상태)
```

**수집 원칙**
- 수집된 원문은 AI 재작성의 원료로만 사용. 게시물에 원문 그대로 발행하지 않음
- Gemini가 완전히 새로 작성 → 저작권 침해 아님 (순수 정보/트렌드는 저작권 보호 대상 아님)
- 각 사이트 이용약관의 자동 수집 금지 조항 사전 확인 권장 (운영자 책임)
- robots.txt 준수

### 14-3. 수집 데이터 구조

```
소스 DB      : sungyesa | girlschannel | cosme
외부 ID      : 소스에서의 원본 게시물 ID
제목         : 원문 제목
내용         : 원문 본문 (텍스트)
소스 URL     : 원본 링크
카테고리     : 소스에서의 카테고리
작성자       : 원문 작성자명
조회수       : 원문 조회수 (인기도 판단)
추가 정보    : JSON (소스별 특수 데이터)
수집 일시    : fetchedAt
소스 날짜    : 원문 작성일
처리 여부    : isProcessed (자동 글쓰기에 사용됐으면 true)
```

### 14-4. 수집 트리거

| 방식 | 설명 |
|------|------|
| 수동 동기화 | 어드민에서 "지금 동기화" 버튼 |
| 자동 스케줄 | 자동 글쓰기 실행 전 자동 수집 |
| 주기적 배치 | 1일 1회 정기 수집 (cron) |

### 14-5. 소스 관리 화면

- 수집된 콘텐츠 목록 (소스/날짜/처리 여부 필터)
- 수동 동기화 버튼
- 소스별 수집 현황 (마지막 동기화 일시, 수집 건수)
- 미처리/처리됨 건수 집계

### 14-6. 소스 추가 정책

향후 소스 추가 시 `external-db.ts`에 소스별 함수 추가.
소스 코드 내 `sourceDb` 문자열만 추가하면 전체 파이프라인에서 인식.
