## 1. 서비스 개요

### 서비스 비전

유튜브 **'렛미인'** 포맷의 변신 스토리 콘텐츠를 기반으로 유입된 팬덤을 자체 앱/웹 플랫폼으로 **락인(Lock-in)** 시키는 프리미엄 뷰티 콘텐츠·상담 플랫폼.

#### 핵심 가치

- **탈(脫) 알고리즘 종속** — 유튜브 알고리즘 의존도에서 벗어나 자체 플랫폼 생태계 구축
- **투명한 정보 제공** — 영상으로 검증된 변신 콘텐츠 + 실제 경험 기반의 신뢰 있는 정보
- **코디네이터 매칭** — 전문 코디네이터가 검토 후 맞춤 프리미엄 병원을 연결

### 핵심 문제 정의

- 성형 시장의 정보 불균형 — 실제 후기보다 병원 마케팅 콘텐츠에 과다 노출
- 커뮤니티 파편화 — 카페·카톡·인스타 등 정보 채널 분산
- 비포&애프터 정보 부족 — 병원 직접 게시 불가(의료광고 규제)로 인한 정보 공백
- 유튜브 → 실행 전환 경로 부재 — 성형 콘텐츠 시청 후 병원 연결까지의 경로가 없음
- 병원 연결 과정의 불투명성 — 가격·시술 정보 비교 어려움

### 목표 사용자

| 구분 | 특성 | 핵심 니즈 |
|------|------|----------|
| 일반 유저 (소비자) | 성형 고민 중인 20–40대 여성 중심 | 신뢰할 수 있는 실제 후기, 맞춤 병원 추천, 커뮤니티 |
| 출연자 (Cast Member) | 렛미인 유튜브 출연 경험자 | 팬 커뮤니티 운영, 스토리 공유, 인플루언서 활동 |
| 병원 (B2B) | 강남권 중심 프리미엄 성형외과, 피부과 | 타겟 고객 확보, 프리미엄 채널 노출 |
| 코디네이터 | 상담 요청 검토 및 병원 매칭 전문 인력 | 효율적 매칭, 매칭 품질 관리 |
| CMS 관리자 | 플랫폼 운영팀 | 콘텐츠 검수, 유저/병원 관리, 출연자 관리 |

### 플랫폼 포지셔닝

**블랙라벨 — 유튜브 렛미인과 연결된 프리미엄 성형 상담 플랫폼**

> 한 줄 정의: "단순 병원 광고가 아닌 '영상으로 검증된 변신 콘텐츠'로 유저의 신뢰를 확보하고, 코디네이터가 엄선한 프리미엄 병원을 매칭하는 콘텐츠-커뮤니티 버티컬 플랫폼"

커뮤니티 + 출연자 스토리가 서비스의 핵심 가치이며, 코디네이터 매칭이 수익화 연결 고리.

### 경쟁 환경 및 차별화

| 기존 플랫폼 | 특징 | 블랙라벨의 차별점 |
|------------|------|-----------------|
| 강남언니 | 병원 리스팅 + 리뷰 + 광고 중심 | 코디네이터 큐레이션 + 유튜브 출연자 검증 콘텐츠 |
| 바비톡 | 커뮤니티 + 병원 정보 | 출연자 팬덤 락인 + 프리미엄 포지셔닝 |
| 여신티켓 | 할인·쿠폰 중심 | VIP 매칭 구조 (소수 프리미엄 병원 집중) |

#### 차별화 근거

1. **신뢰 생성의 주체 변화** — 병원 광고 → 유튜브 영상 기반 검증된 출연자 스토리
2. **프리미엄 큐레이션** — 초기 강남권 프리미엄 병원 약 10여 곳만 세팅, 코디네이터가 맞춤 매칭
3. **지속적 커뮤니티 락인** — 유튜브 팬덤이 출연자 스토리 피드와 상호작용하며 체류
4. **중장기 생태계** — WorkB EMR/CRM 연동, 일본 인플루언서 인프라를 통한 글로벌 확장

### 수익 모델 (미결 — 법률 검토 후 확정)

> 수익 구조는 의료법 제27조 해석에 따라 방향이 달라질 수 있어 법률 자문 후 확정. 개발은 무과금 구조로 먼저 진행.

#### 검토 중인 옵션

| 옵션 | 내용 | 법적 리스크 | 상태 |
|------|------|------------|------|
| 1안: 월정액 입점비 + 광고 | 계약 병원만 매칭 대상, "플랫폼 입점 서비스 이용료" 포지셔닝 | 계약서 설계 필요 | 검토 중 |
| 2안: 광고 배너 단독 | 매칭 무료, 광고 수익만 | 가장 안전 | 검토 중 |
| 3안: 크레딧 시스템 | 병원이 크레딧 충전 후 상담 열람/접근 시 차감 | 의료법 검토 필수 | 검토 중 |
| 4안: 하이브리드 | 월정액 기본 + 프리미엄 노출 추가 과금 | 계약서 설계 필요 | 검토 중 |

#### 공통 주의사항

- 매칭 건당 수수료: **의료법 위반 가능성 높음** — 법률 자문 후 최종 결정

- **어떤 모델이든 출시 전 의료법 전문 변호사 검토 필수**

## 2. 코디네이터 매칭 상담 시스템

> 기능은 포함하되, **수익 모델(병원 과금 구조)은 법률 검토 후 별도 확정**. 현재는 무과금 구조로 개발 진행.
> 의료법 제27조 관련 법률 자문은 출시 전 필수.

### 구조 개요

유저가 시술 고민을 등록하면, 전문 코디네이터가 요청을 검토한 뒤 맞춤 병원 **2~3곳**을 리스트업하여 유저에게 제공한다. 유저는 리스트업된 모든 병원과 채팅이 가능하며, 비교 후 **유저가 직접 결정**한다.

```
유저 상담 접수 → 코디네이터 검토 → 2~3개 병원 리스트업
→ 유저가 리스트 내 모든 병원과 채팅 가능 → 유저가 직접 선택/결정
```

> 플랫폼은 상담 **연결까지만** 담당. 예약 및 시술 결과에 대한 책임 없음.

### 유저 상담 접수 플로우

> 텍스트 직접 입력 최소화, **탭/칩 선택 중심**

#### Step 1 — 시술 카테고리 선택

| 항목 | 내용 |
|------|------|
| UI | 탭 형태 카테고리 (아이콘 + 텍스트) |
| 선택 방식 | 단일 선택 |
| 카테고리 | 성형외과 / 피부과 |
| 세부 | 성형외과: 눈 / 코 / 윤곽(턱·광대) / 가슴 / 지방흡입 / 안면거상 / 기타 |
| | 피부과: 보톡스 / 필러 / 레이저 / 리프팅 / 여드름·흉터 / 제모 / 기타 |
| 관리 | CMS에서 추가·수정·삭제 |

#### Step 2 — 세부 부위 선택

| 항목 | 내용 |
|------|------|
| UI | 칩(Chip) 멀티셀렉 |
| 선택 방식 | 다중 선택 가능, "기타" 선택 시 텍스트 입력창 오픈 |

#### Step 3 — 희망 시술 시기

| 항목 | 내용 |
|------|------|
| UI | 탭 단일 선택 |
| 옵션 | 1개월 이내 / 1~3개월 / 3~6개월 / 미정 |

#### Step 4 — 사진 첨부

| 항목 | 내용 |
|------|------|
| 필수 여부 | 필수 |
| 안내 문구 | "사진을 첨부하시면 더 정확한 맞춤 상담을 받아볼 수 있어요" |
| 업로드 방식 | 갤러리 선택 또는 카메라 촬영 |
| 장수 제한 | 최소 1장, 최대 5장 |
| 이미지 처리 | EXIF 제거, 리사이징 (섹션 8 참조) |
| 공개 범위 | 코디네이터 + 매칭된 병원에만 공개 |

> 부위별 촬영 가이드 프레임 (실루엣 오버레이)은 향후 고도화 버전에서 추가

#### Step 5 — 상담 내용 입력 (선택)

| 항목 | 내용 |
|------|------|
| UI | 자유 텍스트 멀티라인 (선택 입력) |
| 글자 수 | 최대 500자 |
| 안내 | "궁금한 점이나 고민을 자유롭게 적어주세요 (선택)" |

#### Step 6 — 민감정보 수집 동의

> 신체 사진은 개인정보보호법상 민감정보에 해당. 상세는 섹션 6, 10 참조

| 항목 | 내용 |
|------|------|
| 동의 항목 | "민감정보(신체 사진) 수집·이용 동의" (필수) |
| 동의 내용 | 수집 목적, 보관 기간, 제3자(병원) 제공 범위 명시 |
| 미동의 시 | 사진 업로드 불가, 텍스트 전용 요청만 가능 |
| 동의 철회 | 마이페이지에서 언제든 철회 가능 → 기존 사진 즉시 삭제 |

#### Step 7 — 최종 확인 및 제출

| 항목 | 내용 |
|------|------|
| UI | 요약 카드 (시술, 사진 썸네일, 시기) |
| 제출 후 안내 | "블랙라벨 코디네이터가 검토 후 맞춤 병원을 안내해 드립니다" |
| 알림 | 매칭 완료 시 푸시 알림 |

### 배민 스타일 프로세스 UI

유저가 상담 접수 후 [내 상담] 탭에서 확인하는 4단계 진행 UI:

```
① 상담 접수 완료 ✅
② 코디네이터 검토 중 🔄
③ 추천 병원 선정 중 ⏳
④ 병원 매칭 완료 → 채팅 가능 ⏳
```

- 수평 프로그레스 바 + 각 단계 아이콘 애니메이션
- 대기 메시지: "블랙라벨의 전문 코디네이터가 고객님의 상담 내용을 꼼꼼히 검토하여, 가장 적합한 프리미엄 추천 병원을 엄선하고 있습니다"
- 디자인 가이드: 섹션 13 참조

**응답 기준**

| 시간대 | 기준 |
|--------|------|
| 영업시간 내 | 2시간 이내 매칭 완료 |
| 야간/주말 접수 | 다음 영업일 오전 중 안내 |

### 코디네이터 매칭 (CMS)

> CMS 상세는 섹션 7 참조

#### 코디네이터 워크플로우

```
새 상담 요청 알림 수신 → CMS에서 요청 상세 확인 (시술, 사진, 텍스트)
→ 파트너 병원 풀에서 적합한 병원 2~3곳 선정
→ 간단한 매칭 사유 메모 (내부용)
→ 매칭 확정 → 유저에게 "코디네이터가 N개 병원을 추천했습니다" 알림
→ 매칭된 병원에 "새 상담이 매칭되었습니다" 알림
→ 채팅방 자동 개설 (2~3개)
```

#### 매칭 기준

| 기준 | 내용 |
|------|------|
| 시술 전문성 | 요청 카테고리와 병원 전문 분야 일치 |
| 병원 상태 | 현재 활성 상태인 입점 병원만 |
| 응답률 | 기존 상담 응답률이 높은 병원 우선 |
| 분배 균형 | 특정 병원에 과도하게 집중되지 않도록 조정 |

### 유저 — 매칭 결과 확인

매칭 완료 후 유저가 보는 화면:

| 노출 항목 | 설명 |
|----------|------|
| 병원명 + 대표 이미지 | — |
| 전문 시술 분야 | 칩 형태 최대 3개 |
| 병원 소개 | 1줄 미리보기 |
| 병원 상세 보기 | 병원 상세 페이지 (섹션 4) |
| "채팅 시작" 버튼 | 탭 시 1:1 채팅 (섹션 5) |

- 카드 형태로 2~3개 병원 표시
- **모든 병원과 채팅 가능** (1개만 선택하는 구조 아님)
- 유저가 충분히 비교한 뒤 직접 결정

### 병원 — 매칭 알림 수신

코디네이터가 병원을 매칭하면:

| 항목 | 내용 |
|------|------|
| 알림 | "새로운 상담이 매칭되었습니다" 푸시 알림 |
| 확인 가능 정보 | 시술 카테고리, 세부 부위, 희망 시기, 사진 (동의 시), 요청 텍스트 |
| 채팅방 | 자동 개설, 병원이 먼저 인사 메시지 발송 가능 |
| 유저 개인정보 | 채팅 전환 전까지 비공개 |

### 금액 기재 차단

> 채팅에서도 동일 적용 (섹션 5)

- **키워드 필터링**: "만원", "원", 숫자+단위 패턴 감지 시 발송 차단 + 경고 문구 표시
- 필터링 우회 시도(한글 변환 등) 감지 로직 추가
- 위반 적발 시: 경고 1회 → 2회 누적 시 계정 정지
- 신고 처리는 섹션 7 참조

### 엣지 케이스 및 예외 처리

| 상황 | 처리 |
|------|------|
| 코디네이터 영업시간 내 2시간 초과 미매칭 | 에스컬레이션 알림 (CMS 관리자에게), 유저에게 "검토가 지연되고 있습니다" 안내 |
| 코디네이터 24시간 초과 미매칭 | 유저에게 사과 알림 + 우선 처리 배정 |
| 매칭 가능한 병원이 2개 미만 | 1개라도 매칭 가능 시 1개로 진행 + "추가 병원 확인 중" 안내 |
| 유저가 모든 병원과 채팅 후 불만족 | "새로운 요청을 등록해 보세요" 안내 + 재요청 버튼 |
| 재요청 | 동일 내용 재요청 가능. 이전 매칭 병원은 중복 배제 가능 |
| 사진 업로드 실패 | 재시도 버튼 + "네트워크를 확인해 주세요" 안내. 임시 저장 후 이어서 업로드 지원 |
| 유저 상담 취소 | 매칭 전: 즉시 취소. 매칭 후: 채팅방 종료 처리, 병원에 알림 |

### v1.x → v2.0 전환 사항

| 항목 | v1.x (역경매) | v2.0 (코디네이터 매칭) |
|------|-------------|---------------------|
| 구조 | 유저 요청 → 병원이 직접 열람/응답 (5개 선착순) | 유저 요청 → 코디네이터가 2~3개 병원 매칭 |
| 병원 역할 | 요청 열람 + 경쟁 응답 작성 | 매칭 알림 수신 + 채팅 참여 |
| 유저 선택 | 응답 중 1개 선택 → 채팅 | 매칭된 전체 병원과 채팅 → 직접 결정 |
| 대기 UX | 72시간 타이머 | 배민식 4단계 프로세스 UI |
| 유효 기간 | 72시간 자동 만료 | 코디네이터 SLA (영업시간 내 2시간) |
| 사진 업로드 | 부위별 가이드 프레임 | 간소화 (갤러리/카메라) |
| 요청 입력 | 자유 텍스트 필수 (20자 이상) | 탭 선택 중심, 텍스트 선택 입력 |

#### API 변경 사항

| 제거 | 추가 |
|------|------|
| `POST /consultations/:id/respond` (병원 응답) | `POST /admin/consultations/:id/match` (코디네이터 매칭) |
| `GET /consultations/hospital` (병원 요청 열람) | `GET /consultations/:id/matches` (유저 매칭 결과) |
| `POST /consultations/:id/select` (유저 병원 선택) | — |

#### DB 변경 사항

```sql
-- consultation_requests: coordinator 필드 추가
ALTER TABLE consultation_requests
ADD COLUMN coordinator_id BIGINT REFERENCES users(id),
ADD COLUMN coordinator_note TEXT,
ADD COLUMN matched_at TIMESTAMP;

-- 코디네이터 매칭 기록
CREATE TABLE coordinator_matches (
    id BIGSERIAL PRIMARY KEY,
    request_id BIGINT NOT NULL REFERENCES consultation_requests(id),
    hospital_id BIGINT NOT NULL REFERENCES hospitals(id),
    matched_by BIGINT NOT NULL REFERENCES users(id),
    note TEXT,
    status VARCHAR(20) DEFAULT 'matched',
    created_at TIMESTAMP DEFAULT NOW()
);
```

## 3. 성형 커뮤니티

### 비포&애프터 게시판 (MVP-A)

> 유저 자율 커뮤니티로 운영. 플랫폼은 인센티브를 제공하지 않으며, 자연 발생 콘텐츠 구조로 설계. (의료광고 간접 유도 리스크 해소)
> 법적 근거: 섹션 10 — 비포&애프터 간접 광고 리스크

**운영 원칙**
- 유저가 자발적으로 시술 전후 사진 + 후기 게시
- **플랫폼 인센티브 없음** — 포인트·리워드 미제공
- 병원 태깅 허용 (유저 자발적 선택, 강제 아님) → 탭 시 병원 상세로 이동 (섹션 4)
- 태깅된 병원은 해당 게시물 승인·거부 불가 (유저 주도권 유지)
- 신고 기능 → CMS 검수 후 블라인드 (섹션 7)

**게시물 구성**
- 시술 카테고리 태그 (필수)
- 시술 시기 입력 (선택)
- 전후 사진 (선택, 최대 10장) → 공개 버킷 저장 (섹션 8)
- 후기 텍스트 (최소 20자)
- 병원 태깅 (선택)

**초기 콘텐츠 확보 전략**

> 자율 커뮤니티 모델이므로 인센티브 없이 초기 콘텐츠를 확보하는 전략이 필요.

| 전략 | 내용 |
|------|------|
| 시드 콘텐츠 | 운영팀이 실제 시술 경험자 10~20명 섭외하여 초기 후기 게시 요청 (무보상, 순수 자발) |
| 커뮤니티 이전 유도 | 기존 성형 카페·커뮤니티 활동 유저 대상 앱 안내 |
| 유튜브 채널 연동 | 운영사 유튜브 시청자를 앱 커뮤니티로 유도 |
| Q&A 시드 | 운영팀이 자주 묻는 질문 20~30개 사전 등록 |

### 자유 게시판 / Q&A (MVP-B)

- 성형 관련 자유 의견 교환
- 익명 게시 지원
- 태그 기반 분류 (#눈성형, #코성형, #회복기간 등)
- 댓글·대댓글·좋아요 지원

### 투표 / 설문 (MVP-B)

- 유저 또는 운영팀이 투표 생성
- 이미지 포함 투표 지원
- 결과 실시간 공개 (참여 후 노출)
- 설문: 복수 선택형 + 주관식 혼합
- 인기 투표 메인 피드 노출 → 커뮤니티 활성화 유도

### 출연자 스토리 Q&A 피드 *(v2.0 신규)*

> 상세 시스템 명세: 섹션 12 참조

렛미인 유튜브 출연자가 자신의 변신 스토리, 회복 과정, Q&A를 공유하는 전용 피드.

- 유저가 댓글로 질문 → 출연자가 "출연자 답변" 배지와 함께 답변

#### 홈 연동

- 팔로우 시 새 스토리 업로드 푸시 알림

#### Phase

- MVP-A: 출연자 인증/배지만
- MVP-B: 스토리 피드 전체 기능 (팔로우, Q&A, 홈 스토리 바)

## 4. 병원 탐색 + 병원 계정

### 병원 가입 승인 프로세스

```
병원 앱 설치 → 사업자등록번호 입력 → 의료기관 개설 신고증 업로드
→ 병원 기본 정보 입력 → CMS 운영팀 수동 승인 (1~3영업일)
→ 승인 완료 알림 → 병원 계정 활성화
```

| 단계 | 상세 |
|------|------|
| 사업자등록번호 | 국세청 API 연동 유효성 검증 |
| 의료기관 개설 신고증 | 사진 업로드 → CMS에서 수동 확인 (섹션 7) |
| 기본 정보 입력 | 병원명, 주소, 전화번호, 진료시간, 전문 시술 분야, 대표 이미지 |
| 승인 기준 | 사업자번호 유효 + 신고증 확인 + 실제 성형외과 여부 |
| 반려 시 | 사유 안내 + 재신청 가능 |

### 병원 주요 기능

| 기능 | 설명 |
|------|------|
| 코디네이터 매칭 알림 수신 | 매칭 시 푸시 알림 (섹션 2) |

## 5. 1:1 채팅

### 기본 구조

| 항목 | 내용 |
|------|------|
| 기술 스택 | Centrifugo + WebSocket (자체 구축, 섹션 9) |
| 개설 조건 | 코디네이터 매칭 완료 시 자동 개설 (요청 1건당 2~3개 채팅방, 섹션 2) |
| 참여자 | 유저 1 : 병원 1 고정 (요청 1건에 여러 채팅방 동시 운영) |
| 읽음 확인 | 표시 (상대방이 읽은 시각 노출) |
| 인증 | JWT 토큰으로 Centrifugo WebSocket 인증 (섹션 6) |

> v2.0 변경: 유저 선택 시 개설 → 코디네이터 매칭 시 자동 개설. 1건 요청에 2~3개 병원과 동시 채팅 가능.

### 메시지 유형

| 유형 | 내용 |
|------|------|
| 텍스트 | 최대 1,000자 |
| 사진 | 최대 5장/회, 10MB 이하 → 비공개 버킷 (섹션 8) |
| 시스템 메시지 | 채팅 개설 안내 / 종료 안내 / 자동 종료 예고 알림 / 면책 고지 |

### 면책 고지

> 채팅 내 의료 상담 시 비대면 진료 규제 위반 가능성 방지 (섹션 10)

| 항목 | 내용 |
|------|------|
| 노출 시점 | 채팅방 개설 시 시스템 메시지로 자동 표시 |
| 고지 내용 | "이 채팅은 의료 진단·처방이 아닌 상담 안내 목적입니다. 정확한 진단은 대면 진료를 통해 받으시기 바랍니다." |
| 고정 여부 | 채팅방 상단에 항상 고정 배너로 표시 |

### 채팅 종료 조건 — 3가지 모두 적용

| 조건 | 내용 |
|------|------|
| 유저 직접 종료 | "상담 종료" 버튼 → 확인 다이얼로그 → 종료 |
| 상담 완료 처리 | 유저 또는 병원이 "상담 완료" → 양측 확인 후 종료 |
| 자동 종료 | 마지막 메시지 기준 30일 비활성 시 자동 종료 (운영 정책으로 조정 가능) |

종료 후 채팅방은 읽기 전용 전환 (재개설 불가, 기록만 열람)

### 채팅 기록 보관

| 항목 | 내용 |
|------|------|
| 보관 방식 | 종료 후 일정 기간 열람 가능, 이후 자동 삭제 |
| 보관 기간 | **미정** — 개인정보처리방침 수립 시 확정 (섹션 10) |
| 삭제 대상 | 텍스트 + 사진 전체 |
| 유저 알림 | 삭제 N일 전 푸시 알림 발송 |

### 필터링 및 제한

| 항목 | 내용 |
|------|------|
| 외부 연락처 감지 | 전화번호·카카오톡 ID 패턴 감지 시 경고 문구 표시 |
| 금액 기재 차단 | 키워드 필터링 동일 적용 (섹션 2 금액 차단 로직 공유) |
| 신고 | 메시지 단위 신고 → CMS 접수 (섹션 7) |

### 채팅에서 방문 전환

> 외부 연락처를 차단하면서 실제 방문으로 연결하는 경로 설계.

| 항목 | 내용 |
|------|------|
| 방문 예약 안내 | 병원이 "방문 상담 안내" 시스템 메시지를 발송 가능 (병원 주소·진료시간 자동 포함, 섹션 4) |
| 방문 확인 | 유저가 "방문 예정" 버튼 탭 → 날짜·시간 선택 → 양측 확인 (단순 일정 공유, 구속력 없음) |
| 외부 연락처 전환 | 양측 합의 후 유저가 "연락처 공유" 동의 시에만 전화번호 교환 가능 |

## 6. 회원 인증 및 계정 관리

### 일반 유저 가입 플로우

```
앱 실행 → 온보딩 (3장 스와이프) → 카카오 로그인
→ 이용약관 동의 (필수 3건 + 선택 2건)
→ 닉네임 설정 → 관심 시술 선택 (선택) → 메인화면
```

| 단계 | 상세 |
|------|------|
| 온보딩 | 3장 스와이프: (1) 블랙라벨 소개 (2) 코디네이터 매칭 설명 (3) 출연자 스토리 + 커뮤니티 소개 |
| 소셜 로그인 | MVP: 카카오 로그인 단독 / MVP-B: Apple 로그인 추가 |
| 이용약관 동의 | 필수: 서비스 이용약관, 개인정보 수집·이용, 만 18세 이상 확인 |
| 선택 동의 | 마케팅 수신 동의, 민감정보(신체 사진) 수집 동의 |
| 닉네임 | 2~10자, 특수문자 제외, 중복 검사 |
| 관심 시술 | 칩 멀티셀렉 (스킵 가능), 추후 홈 피드 개인화에 활용 |

### JWT 인증 구조

> 기술 상세는 섹션 9 참조

```
카카오 로그인 → Go 서버에서 카카오 토큰 검증
→ 자체 JWT 발급 (access + refresh)
→ Flutter 앱 flutter_secure_storage에 저장
```

| 토큰 | 만료 | 용도 |
|------|------|------|
| Access Token | 15~30분 | API 요청 인증 |
| Refresh Token | 14일 | Access Token 재발급, Redis에 저장 |

- Refresh Token은 **Redis에 저장**하여 강제 로그아웃/탈퇴 시 즉시 무효화
- Centrifugo 접속 시에도 동일한 JWT로 인증 (섹션 5)

### 연령 제한

| 항목 | 내용 |
|------|------|
| 최소 연령 | 만 18세 이상 |
| 확인 방식 | MVP: 자기 신고 체크박스 ("만 18세 이상입니다") |
| 향후 | 본인인증(PASS 등) 도입 시 실제 연령 검증 전환 |
| 미성년자 접근 시 | "만 18세 미만은 서비스를 이용할 수 없습니다" + 가입 차단 |

> 법적 근거: 섹션 10 — 미성년자 보호

> 신체 사진 등 민감정보 포함. 탈퇴 시 완전 삭제 처리 필수 (섹션 10)

| 재가입 | 탈퇴 완료 후 동일 카카오 계정으로 신규 가입 가능 (이전 데이터 복구 불가) |

## 8. 이미지 처리 파이프라인

> 상담 요청 사진, 비포&애프터 사진, 병원 사진 등 모든 이미지에 적용.

### 업로드 처리

```
클라이언트 촬영/선택 → EXIF 메타데이터 제거 (위치정보 등)
→ 클라이언트 사전 압축 (최대 2048px 리사이징)
→ 서버 업로드 (presigned URL 방식)

```

### 서버 후처리

| 처리 | 내용 |
|------|------|
| 리사이징 | 원본 보관 + 썸네일(300px) + 중간(800px) + 풀(2048px) 3종 생성 |

| 포맷 변환 | WebP 변환 (CDN 전송 최적화) |
| 유해 콘텐츠 감지 | MVP: 미적용 / MVP-B: 자동 감지 도입 검토 |

### 저장 구조

| 영역 | 저장 위치 | 접근 권한 | 사용처 |
|------|----------|----------|--------|
| 상담 요청 사진 | 비공개 버킷 | 해당 유저 + 응답 병원만 접근 | 섹션 2 |
| 채팅 사진 | 비공개 버킷 | 채팅 참여자만 접근 | 섹션 5 |
| 비포&애프터 사진 | 공개 버킷 (CDN) | 전체 유저 | 섹션 3 |
| 병원 프로필 사진 | 공개 버킷 (CDN) | 전체 유저 | 섹션 4 |

### 용량 제한

## 9. 기술 스택

### 단계별 인프라 전략

> MVP부터 10만 동접 인프라를 구축하는 것은 과잉 설계. 단계별로 확장한다.

```
Phase 1 (MVP, 0–1만 유저)          Phase 2 (1만–5만 유저)          Phase 3 (5만+ 동접)
─────────────────────────          ────────────────────────          ─────────────────────
Docker Compose + 단일 서버          Docker Swarm / 다중 서버           Kubernetes + HPA
Caddy (리버스 프록시)                Nginx (L7 LB)                    Nginx + K8s LB 이중
PostgreSQL (단일)                   PostgreSQL + PgBouncer            PostgreSQL HA + PgBouncer
Redis (단일)                        Redis Sentinel                    Redis Cluster
Go API (단일 인스턴스)               Go API (2–4 인스턴스)             Go API (HPA 자동 확장)
Centrifugo (단일)                   Centrifugo (단일)                 Centrifugo 클러스터
PostgreSQL LIKE 검색                PostgreSQL trigram                 Elasticsearch
Go goroutine 비동기                 Go goroutine 비동기               RabbitMQ + Worker
Cloudflare R2                      Cloudflare R2                     Cloudflare R2 또는 MinIO
─                                  ─                                 Terraform (IaC)
```

### 클라이언트

| 영역 | 기술 | 비고 |

| 인프라 모니터링 | Prometheus + Grafana | 동접자 수, API 응답시간 |
| 에러 트래킹 | Sentry | Flutter 앱 + Go 서버 |

### 아키텍처 흐름 (MVP)

```
Flutter 앱 (Riverpod)
    │
    ├─ REST API ──→ Caddy (리버스 프록시 + HTTPS)
    │                     │
    │              Go API 서버 (단일 인스턴스)
    │                     │
    │                     ├──→ PostgreSQL
    │                     ├──→ Redis (캐시 + 세션)
    │                     └──→ Go goroutine (푸시/이미지 비동기)
    │
    ├─ WebSocket ──→ Centrifugo ──→ Redis (pub/sub)
    │
    └─ 이미지 요청 ──→ Cloudflare CDN ──→ Cloudflare R2

CMS: Next.js + shadcn/ui
```

### Phase 3 스케일 아웃 아키텍처

```
Flutter 앱 (Riverpod)

    ├─ REST API ──→ Nginx (L7) ──→ K8s LoadBalancer (L4)

인프라 전체: Terraform으로 코드 관리 (IaC)

```

## 12. 출연자 (Cast Member) 시스템

### 개요

유튜브 **'렛미인'** 채널 출연 경험자를 앱 내에서 인증된 출연자(Cast Member)로 관리하고, 출연자의 스토리 피드를 통해 팬덤을 플랫폼으로 락인(Lock-in)하는 시스템.

### 출연자 정의

| 항목 | 내용 |
|------|------|
| 자격 | 렛미인 유튜브 채널에 출연한 경험이 있는 유저 |
| 인증 방식 | CMS 관리자가 수동 승인 (섹션 7) |
| 앱 내 표시 | 닉네임 옆 **"렛미인 출연자"** 인증 배지 |
| 기본 계정 | 일반 유저와 동일 (추가 역할 부여) |

### 출연자 인증 플로우

```
일반 유저로 앱 가입 → 마이페이지에서 "출연자 인증 신청"
→ 출연 회차 정보 + 본인 확인 자료 제출
→ CMS 관리자 검토 (유튜브 영상 대조 확인)
→ 승인 → 출연자 배지 부여 + 스토리 업로드 권한 활성화
```

| 단계 | 상세 |
|------|------|
| 신청 정보 | 출연 회차 (에피소드 번호 또는 URL), 간단한 본인 소개 |
| 확인 방법 | CMS 관리자가 유튜브 영상과 프로필 대조 |
| 승인 소요 | 1~3 영업일 |
| 반려 시 | 사유 안내 + 재신청 가능 |

### 배지 시스템

| 배지 | 조건 | 표시 |
|------|------|------|
| 렛미인 출연자 | 1회 이상 출연 확인 | 닉네임 옆 인증 배지 |

> 향후 확장: 다회 출연, 시리즈 레귤러 등 추가 배지 검토 가능

배지 노출 위치:
- 커뮤니티 게시물 작성자명
- 스토리 피드 프로필
- 댓글 작성자명
- 채팅 (출연자가 상담 요청 시)

### 출연자 프로필

일반 유저 프로필에 추가되는 항목:

| 항목 | 내용 |
|------|------|
| 출연 에피소드 | 유튜브 에피소드 링크 리스트 (CMS에서 등록) |

| 팔로워 수 | 이 출연자를 팔로우한 유저 수 |
| 스토리 피드 | 출연자가 올린 스토리 목록 |

### 팔로우 시스템

| 항목 | 내용 |
|------|------|
| 팔로우 방법 | 출연자 프로필 또는 스토리에서 "팔로우" 버튼 |
| 효과 — 피드 | 팔로우한 출연자의 스토리가 홈 피드 상단 노출 |
| 효과 — 알림 | 새 스토리 업로드 시 푸시 알림 발송 |
| 해제 | 언제든 언팔로우 가능 |
| 마이페이지 | "팔로잉 출연자" 목록 확인 가능 |

### 스토리 포스트

출연자만 업로드할 수 있는 전용 콘텐츠 유형.

#### 스토리 구성

| 항목 | 내용 |
|------|------|
| 텍스트 | 최소 10자 / 최대 1,000자 |
| 사진 | 최대 10장 (섹션 8) |
| 영상 링크 | 유튜브 URL (앱 내 재생 없음, 딥링크) |
| 유튜브 회차 고정 | 출연 에피소드 링크가 스토리 하단에 자동 고정 |

#### 스토리 유형

| 유형 | 설명 |
|------|------|
| 회복 일지 | 시술 후 회복 과정 업데이트 |
| Q&A | 유저 질문에 출연자가 답변 |
| 일상 | 변신 후 일상 공유 |

#### 댓글 Q&A

- 유저가 스토리에 질문 댓글 작성
- 출연자가 답변 시 **"출연자 답변"** 배지 표시
- 출연자 답변은 댓글 상단 고정 옵션

### 홈 피드 노출

```
[출연자A] [출연자B] [출연자C] [출연자D] ...
 (원형)    (원형)    (원형)    (원형)
```

- 원형 프로필 이미지 + 닉네임
- 새 스토리가 있는 출연자: 레드/버건디 테두리 (섹션 13 Accent 컬러)
- 팔로우 중인 출연자 우선 정렬
- 탭 시 해당 출연자의 최신 스토리 피드로 이동

### UX 원칙

#### 1. 최소 입력 (탭 선택 중심)

- 상담 접수: 텍스트 직접 입력 최소화, **탭/칩 선택** 중심
- 구식 신체 아이콘 배제
- 절차 최대한 간소화 (불필요한 Step 제거)

#### 2. 프리미엄 큐레이션 대기 UX

상담 접수 후 대기 구간은 **배민 스타일 프로세스 UI**로 프리미엄 큐레이션 느낌 연출:

```
① 상담 접수 완료 ✅
② 코디네이터 검토 중 🔄
③ 추천 병원 선정 중 ⏳
④ 병원 매칭 완료 → 채팅 가능 ⏳
```

- 수평 프로그레스 바 + 각 단계 아이콘 애니메이션
- 대기 메시지: "블랙라벨의 전문 코디네이터가 고객님의 상담 내용을 꼼꼼히 검토하여, 가장 적합한 프리미엄 추천 병원을 엄선하고 있습니다"
- 응답 기준: 영업시간 내 2시간 이내. 야간/주말은 다음 영업일 오전 중 안내

#### 3. 콘텐츠 이미지 가공

- 유튜브 썸네일을 **그대로 사용하지 않음**
- 플랫폼 톤앤매너에 맞게 **가공하여 노출** (다크 오버레이 + 타이틀 재배치)
- 이미지 위 텍스트는 항상 그림자/그래디언트 처리로 가독성 확보

#### 4. 부드러운 트랜지션

- 화면 전환: fade + slide 복합 트랜지션
- 카드 탭: 미세한 스케일 애니메이션 (0.98 → 1.0)
- 로딩: 스켈레톤 UI (shimmer 효과, 다크 테마에 맞게)
- 스크롤: 히어로 영역 패럴랙스 효과
